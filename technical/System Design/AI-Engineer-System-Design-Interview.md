# AI Engineer 的 System Design 面试怎么考、怎么准备

## 一、核心思路

面试官会给一个模糊的产品需求(比如"设计一个客服 AI 助手"),考察你能不能把它拆成完整的系统架构,并在关键节点上做出有理有据的权衡。

## 二、常见题目类型

### 产品类
- 设计一个 RAG 客服/知识库问答系统
- 设计一个 AI coding assistant(类似 Copilot/Cursor)
- 设计一个多轮对话的 agent(能调用工具、查数据库)
- 设计一个内容审核/分类系统(基于 LLM)
- 设计一个个性化推荐/搜索排序系统(结合 LLM rerank)

### 考察维度(传统 system design + AI 特有层)

1. **需求澄清**:延迟要求、QPS、数据规模、是否需要实时性、准确率 vs 成本的权衡
2. **数据管道**:数据来源、清洗、chunking 策略、embedding 更新频率、增量索引
3. **检索/索引层**:向量库选型(如 pgvector/Milvus/Pinecone)、混合检索(BM25 + 向量)、rerank
4. **模型层**:选哪个模型(大模型 vs 小模型分工)、prompt 设计、fine-tune vs prompt engineering、fallback 策略
5. **服务架构**:流式返回、缓存(prompt cache、语义缓存)、限流、多租户隔离
6. **评估与监控**:离线评估集、在线 A/B、幻觉检测、用户反馈闭环、日志/可观测性
7. **成本与延迟权衡**:token 成本估算、批处理 vs 实时、模型路由(小模型先过一遍再上大模型)
8. **安全性**:prompt injection 防护、PII 过滤、越权访问控制

## 三、怎么准备

1. **打好基础**:传统分布式系统设计(缓存、负载均衡、数据库分片、消息队列)必须扎实,AI system design 是在这基础上叠加 AI 特有组件,不是替代。
2. **吃透 RAG 全链路**:从 chunking 策略到检索评估指标(recall@k、MRR),这是被问得最多的模块,建议能画出完整数据流图并说清每一步的失败模式。
3. **懂成本模型**:能大致估算 token 成本、QPS 下的 GPU/API 调用开销,面试官很看重你是不是"懂钱"而不是纯技术堆砌。
4. **看真实产品架构**:研究一下 Cursor、Perplexity、Notion AI 这类产品公开分享的架构博客,比死记概念更有说服力。
5. **练习画图讲故事**:用"需求澄清 → 高层架构 → 深入 2-3 个模块 → 权衡讨论"的固定节奏来练习,45 分钟内讲完一个完整方案。
6. **准备失败案例**:面试官常追问"如果检索出来的内容是错的怎么办""如果模型输出不稳定怎么处理",提前想好几个真实的失败模式和对应缓解方案会很加分。
