# AI Engineer System Design 常见题目拆解

配合 [AI-Engineer-System-Design-Interview.md](./AI-Engineer-System-Design-Interview.md) 使用。每道题按"需求澄清 → 高层架构 → 关键模块深入 → 权衡与失败模式"的节奏拆解。

---

## 题目一:设计一个 RAG 客服/知识库问答系统

### 需求澄清
- 知识库规模:文档数量级(千 vs 百万)、更新频率(每天 vs 实时)
- 延迟要求:是否需要流式返回,P95 延迟目标
- 准确率要求:能否容忍幻觉,是否需要引用来源
- 多租户:是否要按客户隔离知识库
- QPS:并发用户量级

### 高层架构
```
用户问题 → 查询理解/改写 → 检索(向量 + 关键词) → Rerank → 拼接上下文 → LLM 生成 → 流式返回 + 引用来源
                                    ↑
                            离线:文档摄入 → 清洗 → chunking → embedding → 写入向量库
```

### 关键模块深入

1. **Chunking 策略**
   - 固定长度 vs 语义分段(按段落/标题切分)
   - chunk 大小与召回率、上下文利用率的权衡(太小丢上下文,太大稀释相关性)
   - 保留 metadata(来源、章节、时间)方便过滤和引用

2. **检索层**
   - 向量检索(embedding 相似度)+ BM25 关键词检索做混合检索,弥补纯向量检索对专有名词/精确匹配不敏感的问题
   - 向量库选型:pgvector(已有 Postgres 基建时省心)、Milvus/Qdrant(大规模、高性能场景)
   - Top-K 召回后用 rerank 模型(如 cross-encoder)二次排序,提升精度

3. **生成层**
   - Prompt 设计:明确要求"仅基于提供的上下文回答""不确定时说不知道",降低幻觉
   - 引用来源:让模型在回答中标注引用的 chunk,便于用户验证
   - 模型选择:简单问题用小模型/规则先拦截,复杂问题才调大模型(成本优化)

4. **评估**
   - 离线:构建 QA 评估集,衡量 recall@k、答案相关性(可用 LLM-as-judge)
   - 在线:用户反馈(点赞/点踩)、人工抽检幻觉率

### 权衡与失败模式
- **检索到的内容不相关**:加 rerank,或设置相关性阈值,低于阈值时明确告知用户"未找到相关信息"而不是硬编译答案
- **知识库更新滞后**:增量索引 pipeline,对高频变动内容(如政策)优先处理
- **多轮对话上下文丢失**:需要在检索阶段考虑对话历史,做查询改写(query rewriting)避免代词指代不清
- **成本失控**:限制上下文长度,缓存高频问题的答案(语义缓存)

## 题目二:设计一个 AI Coding Assistant(类似 Copilot/Cursor)

### 需求澄清
- 场景范围:补全(inline completion)vs 对话式编辑(chat-based edit)vs agentic 多文件修改
- 延迟要求:补全场景要求 <100-200ms,对话场景可以放宽到秒级
- 上下文来源:当前文件、整个仓库、跨文件依赖关系
- 是否需要执行代码(测试、编译)来验证生成结果

### 高层架构
```
IDE 插件 → 上下文收集(当前文件/光标位置/相关文件) → 检索仓库上下文(embedding/AST) 
        → Prompt 构造 → 模型推理(流式) → 客户端渲染 → 用户接受/拒绝反馈
                                                              ↓
                                                    反馈用于模型/prompt 迭代
```

### 关键模块深入

1. **上下文收集与仓库理解**
   - 静态分析:AST 解析、符号索引(函数/类定义跳转关系),辅助召回相关代码片段
   - 检索:对仓库代码做 embedding 索引,结合最近编辑历史、打开的文件做上下文拼接
   - 上下文窗口有限,需要做优先级排序(当前文件 > 直接依赖 > 全局检索结果)

2. **补全场景的低延迟设计**
   - 用小模型/本地模型做首屏快速补全,大模型做更复杂的多行生成,分层调用
   - 客户端做请求防抖(debounce)和取消(用户继续输入时取消旧请求)
   - 边缘缓存/预测性预取,减少往返延迟

3. **Agentic 多文件修改场景**
   - 需要工具调用能力:读文件、写文件、执行终端命令、跑测试
   - 需要执行结果反馈闭环:改完代码后跑测试,失败则让模型看错误信息重试(self-repair loop)
   - 需要人工确认关卡(尤其是删除文件、执行 shell 命令等高风险操作)

4. **评估**
   - 接受率(acceptance rate)、编辑距离(生成后用户又改了多少)
   - 端到端任务完成率(agentic 场景:任务是否真的跑通)

### 权衡与失败模式
- **生成代码引入 bug**:接入测试/lint/类型检查作为生成后的验证层,失败自动重试
- **上下文过大导致延迟高**:做上下文压缩/摘要,或用更便宜的模型先筛选相关文件
- **模型幻觉出不存在的 API**:结合仓库真实符号表做约束/校验,生成后做静态检查
- **多文件修改冲突**:需要事务性思路,修改前生成 diff 预览,允许用户逐个 review

## 题目三:设计一个多轮对话的 Agent(能调用工具、查数据库)

### 需求澄清
- Agent 的任务边界:能调用哪些工具(查库、发邮件、调用外部 API)
- 是否需要长期记忆(跨会话记住用户偏好)vs 仅会话内短期记忆
- 是否允许 agent 自主执行高风险操作(如下单、转账),还是必须人工确认
- 并发会话量级、单次任务的最大执行时长/步数限制

### 高层架构
```
用户输入 → 会话管理(短期记忆/历史) → 意图理解/规划(Planner) 
        → 工具调用(Tool Router → 具体工具:DB查询/API/搜索) → 观察结果 
        → 判断是否完成任务(继续规划 or 生成最终回复) → 返回用户
                    ↓
            长期记忆存储(用户偏好、历史任务摘要)
```

### 关键模块深入

1. **规划与工具调用(ReAct / function calling 模式)**
   - 模型输出结构化的工具调用请求(而不是自由文本),用 JSON schema 约束
   - 每一步:模型决定"调用工具"还是"直接回复",循环直到任务完成或达到步数上限
   - 工具调用失败(超时/报错)需要让模型看到错误信息并决定重试或换策略

2. **记忆管理**
   - 短期记忆:会话历史,超过窗口时做摘要压缩而不是简单截断
   - 长期记忆:结构化存储用户偏好/历史任务结果,按需检索注入(类似 RAG,但检索对象是记忆而非文档)

3. **安全与权限控制**
   - 工具调用需要按用户身份做鉴权(agent 不能越权访问其他用户数据)
   - 高风险操作(转账、删除数据)设置人工确认关卡,或者限制在只读工具范围内自主执行
   - 防止 prompt injection:如果工具返回的内容(如网页/邮件正文)可能包含恶意指令,需要对工具输出做隔离,不能让其直接覆盖系统指令

4. **可控性与终止条件**
   - 设置最大步数/最大工具调用次数,防止死循环
   - 加入"是否已经完成任务"的显式判断步骤,避免 agent 无限试探

### 权衡与失败模式
- **无限循环调用工具**:硬性步数上限 + 超时熔断,达到上限时降级为向用户说明卡在哪一步
- **工具调用参数错误**:用 JSON schema 强约束 + 校验层,校验失败时把错误信息反馈给模型重试
- **多个 agent/工具并发访问同一资源**:需要考虑幂等性(如重复下单)和资源锁
- **长任务的可观测性**:需要记录每一步的思考/工具调用/结果,方便调试和审计(不能是黑盒)

## 题目四:设计一个内容审核/分类系统(基于 LLM)

### 需求澄清
- 审核对象:文本/图片/视频,单一模态还是多模态
- 延迟要求:是否要在发布前实时拦截(同步)还是发布后异步复审
- 分类粒度:二分类(违规/不违规)还是多标签(暴力、色情、垃圾广告等细分类别)
- 误判成本:漏判(违规内容漏放)和误判(正常内容被拦)哪个代价更高,决定阈值设定方向
- QPS 与内容规模:每天审核量级

### 高层架构
```
内容发布 → 规则/黑名单快速过滤(便宜、低延迟) → 小模型初筛(分类器) 
        → 命中风险阈值区间 → LLM 精细审核(给出理由和分类标签) 
        → 高风险 → 自动拦截 / 中风险 → 人工复审队列 / 低风险 → 放行
                                                ↓
                                    人工复审结果回流,用于优化规则和模型
```

### 关键模块深入

1. **分层过滤(漏斗式设计)**
   - 第一层:规则引擎(关键词黑名单、正则),成本几乎为零,拦截明显违规内容
   - 第二层:轻量分类模型(传统 ML 或小型 LLM),快速给出风险分数
   - 第三层:只对风险分数落在"不确定区间"的内容调用大模型精细审核,兼顾成本和准确率
   - 这种漏斗设计是控制成本的关键:大部分内容不需要走到最贵的一层

2. **同步 vs 异步审核**
   - 同步(发布前拦截):延迟敏感,只能用便宜快速的层级,大模型审核放到异步复审
   - 异步(发布后复审):可以用更贵更准的模型,发现问题后下架+通知

3. **Prompt/分类设计**
   - 要求模型输出结构化结果(类别 + 置信度 + 理由),不要只输出"是/否",理由便于人工复审时快速验证
   - 针对边界案例(讽刺、反讽、教育科普 vs 真实违规)专门构造评估集,LLM 在这类场景容易误判

4. **人工复审闭环**
   - 中风险内容进入人工复审队列,人工标注结果用于:更新规则黑名单、作为微调数据、调整阈值
   - 复审优先级排序:高曝光内容优先复审

### 权衡与失败模式
- **模型误判导致正常内容被拦(误伤)**:设置申诉通道,记录误伤率作为核心监控指标,不能只看拦截率
- **审核规避(对抗性内容,如变形文字、隐喻)**:需要定期用红队测试更新规则和评估集,单靠一次训练无法长期有效
- **人工复审队列积压**:按风险分数和内容曝光量做优先级排序,而不是简单 FIFO
- **审核标准随政策变化**:分类体系和 prompt 需要可配置化,不能硬编码在代码里,方便运营快速调整

## 题目五:设计一个个性化推荐/搜索排序系统(结合 LLM rerank)

### 需求澄清
- 场景:搜索排序(有明确 query)还是推荐(无明确 query,基于用户画像)
- 延迟要求:搜索场景通常要求 <200-500ms 端到端
- 数据规模:候选集大小(百万级 item)、用户量级
- LLM 在整个链路中扮演什么角色:精排阶段的 rerank,还是用于生成查询理解/用户意图摘要
- 冷启动问题:新用户/新 item 如何处理

### 高层架构
```
用户 query/行为 → 查询理解(意图识别、query 改写) 
                → 召回(多路召回:协同过滤 + 向量召回 + 关键词召回) 
                → 粗排(轻量模型,候选集从万级降到百级) 
                → 精排(LLM/复杂模型 rerank,百级降到十级) 
                → 重排/多样性调整(去重、打散、业务规则) 
                → 展示 + 用户行为反馈回流
```

### 关键模块深入

1. **多路召回**
   - 传统召回(协同过滤、基于规则)+ 向量召回(embedding 相似度,捕捉语义相关性)
   - LLM 可以用于生成"查询扩展"(把用户 query 改写成多个语义相近的查询),提升召回覆盖率
   - 多路召回结果去重合并,进入粗排

2. **分层排序架构(为什么不能全用 LLM)**
   - 候选集从百万级降到万级:必须用便宜的模型(向量检索/简单打分),LLM 推理成本太高不可能覆盖全量
   - 粗排(万级→百级):轻量模型,兼顾速度和效果
   - 精排(百级→十级):这一层才用 LLM rerank,输入变小,可以承受更高延迟和成本
   - 核心原则:越靠前的阶段,候选集越大,模型必须越便宜;越靠后,候选集越小,才用得起大模型

3. **LLM Rerank 的具体做法**
   - 把用户上下文(历史行为摘要、当前意图)和候选 item 的关键信息拼进 prompt,让 LLM 打分或排序
   - 候选数量太多时无法一次性塞进 context,需要分批打分或用 pairwise/listwise 的排序策略
   - 需要考虑 LLM 推理的延迟波动,做好超时降级(超时则回退到传统排序模型的结果)

4. **冷启动与多样性**
   - 新用户:用 LLM 基于注册信息/首次行为快速生成初始画像,弥补协同过滤数据不足
   - 新 item:基于内容(标题/描述)做 embedding,先靠内容相似度获得曝光,再用真实反馈迭代
   - 多样性:精排结果不能只看相关性分数,要做类别打散、去重,避免结果同质化

### 权衡与失败模式
- **LLM rerank 延迟过高拖累整体响应**:设置超时阈值,超时自动降级为传统排序模型结果,不能让用户等待
- **LLM 打分不稳定(同样输入不同次打分不一致)**:适当降低 temperature,或用多次采样投票平滑,关键场景避免完全依赖单次 LLM 输出
- **头部效应(热门内容一直被推荐,新内容没有曝光机会)**:引入探索机制(如 bandit 算法)在排序中预留一定比例的探索位
- **成本失控**:精排候选集必须严格控制在百级以内才调用 LLM,不能让 LLM 处理未经过筛选的大候选集

## 通用回答节奏建议

1. 澄清需求(2-3 分钟):不要跳过,问延迟/规模/准确率/成本这几个维度
2. 画高层架构图(5 分钟):先给出端到端数据流,再说"我想深入讨论 XX 模块"
3. 深入 2-3 个模块(20-25 分钟):面试官通常会引导你深入某个方向,顺着走
4. 讨论权衡与失败模式(5-10 分钟):主动抛出"如果 XX 出错怎么办",展示工程判断力
