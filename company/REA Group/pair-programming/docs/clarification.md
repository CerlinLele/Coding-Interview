# 面试中需要澄清的点

## TDD（测试驱动开发）
面试官不强制要求特定的开发方式，以下都可以接受：
1. **先实现后验证** - 写完代码后运行测试确认功能正常，完全可以
2. **先写测试后实现** - 如果这是你习惯的方式，也完全没问题
3. **增量开发边写边测** - 逐步实现并同步测试，是个很好的方法

**建议**：选择你最自然、最流畅的方式。面试官更关注你的思路和协作，而不是固守某种开发模式。

## Git Commit（版本提交）
需要在面试开始时澄清：
- **是否需要在编码过程中定期 commit？**
- **还是只需要确保代码能跑通即可？**

**可能的场景**：
- 如果需要 commit：建议在每个小功能完成并测试通过后提交，commit message 简洁说明改动即可（例如 "Add JUMP command" 或 "Handle boundary for new direction"）
- 如果不需要 commit：专注编码和测试即可，不用分心处理 git 操作

**建议开场问法**：
"对了，在编码过程中，你们希望我定期做 git commit 吗？还是说只要代码最后能跑通就行？"

## 其他可能需要澄清的点（视情况询问）

### 代码风格
- 是否有特定的代码风格偏好？（例如变量命名风格、注释详细程度）
- 如果现有代码库有明确风格，直接延续即可

### 扩展功能的完整度
- 新功能是否需要完整的错误处理和边界检查？
- 还是先实现 happy path，再根据时间补充？

**策略**：通常建议先让核心功能跑起来，再逐步加固。编码时口头说明："我先让基本功能跑通，然后我们可以加上边界检查。"

## 设计决策不确定时的应对策略

面试中常会遇到 trade-off 难以判断的情况——不确定该选 A 还是 B。**这是正常的，可以用以下流程处理**：

### ✅ 好的做法

**1. 快速做决策，但说出理由**

```
"我倾向选择 B，主要理由是：
 - 符合单一职责原则（Table 管理自己的数据结构）
 - 易于后续维护和扩展
 
缺点是需要多一次方法调用。

你觉得这个方向对吗？或者有其他考虑？"
```

**2. 邀请反馈和讨论**

不要装作完全确定。面试官看重的是：
- 你能识别 trade-off 吗？
- 你能接纳建议吗？
- 你的推理逻辑清晰吗？

**3. 根据反馈灵活调整**

```
面试官: "嗯，不过如果将来表超大了呢？"
你: "哦，好点子。那可能我们需要根据表大小动态选择存储结构...
     或者现在就假设中等大小，用混合存储？"
```

**4. 达成共识后继续**

```
"好的，我们就用这个方案。这样 Robot 调用 
table.update_robot_position()，Table 维护自己的网格。
下一步我们开始实现..."
```

### ❌ 要避免的做法

| 不要做 | 为什么 | 后果 |
|--------|--------|------|
| 完全犹豫不决 | 没有判断力 | 浪费时间，无法推进 |
| 选了却说不出理由 | 显得随意 | 面试官不信你有思考 |
| 被问出漏洞后死守方案 | 不够灵活 | 显得固执，难合作 |
| 中途反复改主意 | 计划性差 | 显得不专业 |
| 假装很确定 | 不诚实 | 一旦实现出问题，信任度下降 |

### 面试官心态

好的面试官会：
- **主动指点**：如果你的方案有问题，会给提示或反问
- **考察灵活性**：你能接纳反馈吗？会改主意吗？
- **看思考深度**：你对 trade-off 有认识吗？
- **评估沟通能力**：你如何解释和讨论设计选择？

**重点**：面试不是考你有没有完美答案，是考你怎么在有限信息下做决策、怎么跟团队沟通、怎么根据反馈改进。

### 推荐的对话框架

```
【陈述方案】
"我倾向选 A，因为..."

【承认不确定】
"我还没完全确定，但基于 X 和 Y，我觉得 A 比较合理。"

【邀请反馈】
"你有其他想法吗？我是否遗漏了什么？"

【灵活调整】
"哦，我没想到那一点。那我们改成这样是否更好...?"

【确认方向】
"好的，我们就这样继续实现。"
```

这个流程展示了：
- ✅ 清晰的设计思路
- ✅ 对 trade-off 的认识
- ✅ 谦虚和开放的态度
- ✅ 有效的沟通和协作能力
