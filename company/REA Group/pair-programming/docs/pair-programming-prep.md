## 面试基本信息

REA 明确说明会从三个维度评估：

**重点**：面试官明确表示"不会评估你对所选语言的深度掌握"，而是看你如何解决问题、如何协作。

## 技术准备清单

### 环境确认（面试前必查）

- [ ] GitHub 账号可以正常访问并克隆私有仓库
- [ ] Python 3 环境可用（`python3 --version`）
- [ ] 能够运行 pytest 或 unittest（`pip list | grep pytest`）
- [ ] 编辑器启动流畅（VS Code / PyCharm / Cursor）
- [ ] Zoom 屏幕共享测试过，代码清晰可读
- [ ] 网络稳定，备用热点已准备

### Python 基础复习

虽然不考语言本身，但你需要流畅表达思路，建议快速过一遍：

- 类和对象的基本定义
- 列表、字典、元组的常见操作
- 函数定义、参数传递、返回值
- 简单的文件读写或命令行参数解析
- 异常处理的基本写法
- 如何写和运行单元测试（pytest / unittest）

### Toy Robot 题型预习

"Toy Robot" 是经典的 OOP 练习题，通常包含：

**核心功能**：
- Robot 在一个网格（例如 5x5）上移动
- 支持命令：`PLACE X,Y,FACING`（放置机器人），`MOVE`（向前一步），`LEFT`/`RIGHT`（转向），`REPORT`（报告当前位置和朝向）
- 边界保护：机器人不会掉出网格

**常见扩展方向**：
- 添加新命令（例如 `JUMP`、`BACKWARD`）
- 支持多个机器人
- 添加障碍物
- 改变网格形状或大小
- 命令历史和撤销/重做
- 从文件读取命令序列

**设计要点**：
- 状态封装（位置、朝向）
- 命令解析和验证
- 边界检查逻辑
- 可扩展性（新增命令不应大改现有代码）

## Pair Programming 最佳实践

### 沟通节奏

1. **先理解，再动手**
   - 读完题目和现有代码后，先口头总结你的理解
   - 确认需求边界："我理解的是……对吗？"
   - 明确扩展目标："我们今天要加的是 X 功能，对吧？"

2. **边写边说**
   - 写代码时要说出你的思路："我现在要添加一个方法来处理……"
   - 遇到选择时说出权衡："我可以用 A 方案或 B 方案，A 更简单但……"
   - 不要长时间沉默敲代码

3. **主动求助**
   - 卡住时不要硬撑："我在想这个边界情况该怎么处理，你有建议吗？"
   - 把面试官当队友，不是考官

### 编码习惯

1. **增量开发**
   - 先写最小可工作版本（hardcode 都可以）
   - 确认逻辑正确后再重构
   - 每完成一个小步骤就运行测试

2. **测试驱动**
   - 看现有代码有没有测试文件，如果有就延续测试风格
   - 新功能至少要能跑通一个简单用例
   - 边界情况（如越界、非法输入）要口头提及，时间允许再补测试

3. **代码清晰度**
   - 变量命名要有意义（`new_x` 优于 `nx`）
   - 函数职责单一（一个函数只做一件事）
   - 避免嵌套过深（提前 return，提取子函数）

### 常见陷阱

- **过度设计**：不要一开始就搞工厂模式、策略模式，先用最简单的 if-else
- **沉默写代码**：面试官看不到你脑子里在想什么，一定要说出来
- **忽略现有代码**：先读懂别人写的结构，不要推倒重来
- **不跑测试**：写完不验证就继续写下一段，容易累积错误

## 针对你的背景的建议

### 你的优势

根据你的简历和之前的电话面试：

1. **实战经验强**
   - 你做过 agentic workflow orchestration（LangChain/LangGraph）
   - 熟悉 observability、schema validation、PII handling
   - 有生产环境支持经验（Wrkr 的 SuperStream 系统）

2. **平台思维**
   - REA 强调 "platform capability" 和 "reusable components"
   - 你在 AceMyJob 做的内部工具和 API 设计直接对应这个要求

3. **Pragmatism 天然符合**
   - 你的项目都是以"能用、能扩展、能观测"为目标，不是学术 demo

### 如何展现这些优势

在 pair programming 中，适时提到（但不要刻意炫耀）：

- **谈代码组织时**："在我之前的项目里，我们会把命令解析和业务逻辑分开，这样测试更容易"
- **谈错误处理时**："我习惯先确保 happy path 能跑通，然后再加 validation，这样调试更快"
- **谈扩展性时**："如果以后要加更多命令，我们可以考虑一个命令注册表，但现在先用简单的 if-else"

### 需要注意的点

1. **不要过度 AI 化**
   - 这是个基础编程题，不要往 LLM、agent、orchestration 上扯
   - 专注展现基本的 OOP、测试和重构能力

2. **控制节奏**
   - 90 分钟看似很长，但扣掉破冰、理解需求、最后总结，实际编码时间不到 60 分钟
   - 不要陷入细枝末节（比如花 10 分钟纠结变量命名）

3. **Learning Mindset 的展现**
   - 如果面试官提出不同思路，不要急于辩护，先说"哦这个角度不错，我之前没想到"
   - 结束时可以问："如果是你来做这个扩展，你会用什么结构？"

## 面试当天流程建议

### 开始 5 分钟

1. 快速自我介绍（30 秒版本，提到你当前在做 AI platform 工作）
2. 让面试官介绍他们各自在 REA 做什么（显示你对团队的兴趣）
3. 确认屏幕共享清晰度："你们能看清代码吗？"

### 理解需求 10 分钟

1. 克隆代码库，快速浏览目录结构
2. 读 README（如果有）
3. 运行现有测试，确认环境 OK
4. 口头总结："现在代码库支持 X、Y、Z，我们要加的是 A 功能，对吧？"
5. 问一两个澄清问题（不要问太多）

### 编码 50-60 分钟

1. **第一步**：写一个最简单的测试用例（或直接在 main 里试跑）
2. **第二步**：实现最小可工作版本
3. **第三步**：处理边界情况
4. **第四步**：（时间允许）重构和补全测试

中间每 15 分钟左右问一次："我们现在进度 OK 吗？"

### 收尾 10 分钟

1. 快速演示你的实现："我们来跑一下，现在机器人可以……"
2. 口头说明已知的局限："如果时间更多，我会加上 X 和 Y"
3. 问问题（准备 2-3 个，例如）：
   - "REA 的工程师日常也会做这种 pairing 吗？"
   - "Voyager 团队现在最大的技术挑战是什么？"
   - "从你们的角度看，这个岗位 30 天内最重要的交付是什么？"
