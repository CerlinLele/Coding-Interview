## 面试框架

### REA 的三大评估维度

**1. Current Technical Knowledge** — 工程技能和技术基础
- 能否快速理解现有代码结构
- 能否识别合适的数据结构和算法
- 代码组织和命名是否清晰
- 是否考虑边界情况和错误处理

**2. Pragmatism** — 简单优于复杂
- 能否用最小代码实现功能（不过度设计）
- 是否能够做出权衡决策（成本 vs 收益）
- 是否能够有效利用时间（不纠缠细节）
- 是否重视现实约束（时间、可维护性）

**3. Learning Mindset** — 开放学习和协作
- 能否主动提问澄清需求
- 是否接纳面试官的建议和反馈
- 能否解释自己的设计思路
- 是否愿意讨论多个方案的权衡

### 90 分钟时间分配

| 阶段 | 时间 | 活动 |
|-----|------|-----|
| 破冰和自我介绍 | 5 分钟 | 简短自我介绍，了解面试官背景 |
| 理解需求 | 10 分钟 | 克隆代码，跑测试，确认任务范围 |
| Level 1 扩展 | 10-15 分钟 | 基础命令扩展（BACKWARD/JUMP） |
| Level 2 扩展 | 15-20 分钟 | 状态管理（UNDO/REDO） |
| Level 3 扩展 | 15-20 分钟 | 多实体或复杂交互（可选） |
| 收尾总结 | 10 分钟 | 演示、说明局限、提问 |

## 按难度递进的问题

### 🟢 Level 1：基础命令扩展（10-15 分钟）

#### 典型题目

```
"能不能给机器人加一个 BACKWARD 命令？"
"能不能实现 JUMP X,Y 功能，让机器人直接跳到目标位置？"
```

#### 面试官可能的追问序列

1. **初始问题**
   - "能否在 5-10 分钟内实现 BACKWARD？"
   - 考察：快速理解现有代码、复用逻辑的能力

2. **边界情况追问**
   - "如果机器人在 (0,0) 面向北，BACKWARD 会怎样？"
   - 考察：是否考虑边界条件、是否运行测试验证

3. **代码质量追问**
   - "我看你 BACKWARD 和 MOVE 有很多重复代码。能优化吗？"
   - 考察：DRY 原则、重构能力、是否愿意改进

4. **深度思考追问**
   - "如果以后要加 DIAGONAL（斜向），现在的设计还够用吗？"
   - 考察：对可扩展性的考虑、是否过度设计还是完全不考虑

#### 你应该说的

✅ **好的回答框架**：
```
1. 先读现有代码
   "我看 move() 使用了 DIRECTION_DELTAS 字典，BACKWARD 
    就是反向移动一步。"

2. 说出设计选择
   "我有两个选项：
    - 复制 move() 逻辑但 delta 取反（快但重复）
    - 给 move() 加个参数 direction='forward|backward'（更清晰）
    
    我选第二个，因为这样修改边界检查时只需改一处。"

3. 实现并验证
   "写完了，我跑一下测试看看……"
   [运行测试或手动验证边界情况]

4. 对反馈开放
   面试官："这样会有个问题……"
   你："哦，我没想到，那我改成……"
```

❌ **要避免的做法**：
- 沉默敲代码 10 分钟，最后说"应该能用"（无法展示思路）
- "我不知道怎么优化，先这样吧"（不够主动改进）
- 坚守第一版设计"这样已经够用了"（不够灵活）

### 🟡 Level 2：状态管理扩展（15-20 分钟）

#### 典型题目

```
"能不能实现命令历史和 UNDO 功能？"
"现在有 UNDO 了，能加上 REDO 吗？"
```

#### 面试官可能的追问序列

1. **初始问题**
   - "怎么设计 UNDO？你会怎么存储状态？"
   - 考察：能否说出多种方案（快照 vs 反向操作）和权衡

2. **设计选择追问**
   - "UNDO 一个 MOVE 应该回退 1 格还是恢复整个操作？"
   - "如果有 ROTATE 命令，UNDO 后 move_count 应该变吗？"
   - 考察：对状态一致性的理解

3. **边界情况追问**
   - "连续 UNDO 多次会怎样？到最后还有 history 吗？"
   - "UNDO 一个未放置的机器人会怎样？"
   - 考察：全面性和防御性编程

4. **扩展追问**
   - "现在要加 REDO，双栈方案怎么处理'执行新命令后 redo 栈清空'的问题？"
   - 考察：对栈的理解、能否识别设计陷阱

#### 你应该说的

✅ **好的回答框架**：
```
1. 说出权衡
   "UNDO 有两种实现方式：
    - 存储状态快照（历史记录 (x,y,facing)）：简单但内存占用
    - 存储反向操作：省内存但要实现每个命令的 undo 版本
    
    我选方案 1，因为我们不知道后续会加多少命令，
    快照更简单也更安全。"

2. 说出关键设计
   "UNDO 返回一个完整的 (x, y, facing, move_count) 状态，
    因为我们需要同时恢复位置和移动计数。"

3. 说出已知局限
   "目前如果 UNDO 多次到没有历史，会返回失败。
    这是合理的行为，但如果需要也可以加上。"

4. 对反馈响应
   面试官："REDO 呢？"
   你："REDO 需要单独的栈。关键是任何新 MOVE/LEFT/RIGHT 
       都要清空 redo 栈，因为我们进入了新的时间线。"
```

❌ **要避免的做法**：
- 完全犹豫不决"我还没想好"（花太多时间没决策）
- "我选快照方案"但说不出为什么
- 被问"REDO 后 move_count 怎么算"时慌张或防守

### 🔴 Level 3：多实体或复杂交互（20-30 分钟）

#### 典型题目

```
"现在支持多个机器人在同一个桌面上。"
"机器人不能相互碰撞，也不能进入障碍物格子。"
"如果机器人 A 推动机器人 B，B 能推动 C 吗？"
```

#### 面试官可能的追问序列

1. **架构问题**
   - "Table 应该如何跟踪多个机器人的位置？"
   - "碰撞检查应该放在 Robot 还是 Table？"
   - 考察：职责划分、对象设计的理解

2. **API 设计问题**
   - "怎么初始化多个机器人？`PLACE 1,1,NORTH ROBOT1`？"
   - "REPORT 输出多个机器人的信息吗？"
   - 考察：对向后兼容性和 API 易用性的思考

3. **一致性问题**
   - "机器人 A 和 B 同时 MOVE 到同一格子怎么办？"
   - "如果 A 推 B，B 推 C，最后 C 推不动，A 应该停在哪？"
   - 考察：对原子性和事务的理解

4. **性能问题（如果时间充裕）**
   - "桌子是 100x100，有 50 个机器人，碰撞检查怎么优化？"
   - 考察：是否考虑复杂度

#### 你应该说的

✅ **好的回答框架**：
```
1. 先画草图
   "让我简单画一下……现在 Table 需要管理多个 Robot 对象，
    我用一个字典或二维数组存储位置信息。"

2. 设计职责
   "碰撞检查放在 Table.is_valid_position()，因为 Table 
    是真理来源——它知道谁在哪儿、哪儿有障碍。
    Robot 的 move() 方法调用 Table 来验证。"

3. 增量实现策略
   "我先把单个机器人的碰撞检查做通，验证后再加多机器人场景。"

4. 对限制坦诚
   "如果要支持同时移动和冲突解决，需要更复杂的事务机制。
    现在我假设命令是序列执行的。"
```

❌ **要避免的做法**：
- 一上来就说"我用消息队列和事件驱动"（过度设计）
- 没想清楚就开始写代码，陷入 bug
- 被问到推挤链式（A推B推C）时完全懵

---

### 🔴 Level 4：架构重构或开放讨论（如果时间充裕）

#### 典型题目

```
"能不能从文件读取一系列命令执行？"
"命令现在有 7 个，如果以后到 50 个，代码该怎么组织？"
"如果要支持 undo/redo 的完整重放，怎么设计？"
```

#### 面试官可能的追问序列

1. **何时抽象的问题**
   - "现在用 if-elif 很清晰，什么时候你会考虑重构？"
   - 考察：对"YAGNI"（You Aren't Gonna Need It）的理解

2. **序列化问题**
   - "怎么保存当前的游戏状态到磁盘？"
   - "uuid 怎么序列化到 JSON？"
   - 考察：对持久化和格式的思考

3. **开放式设计问题**
   - "如果支持 100 个机器人和自定义大小的桌子，架构怎么改？"
   - 考察：系统设计能力、是否能识别可扩展点

#### 你应该说的

✅ **好的回答框架**：
```
1. 守住务实底线
   "现在用 if-elif 其实很清晰，没必要抽象。如果命令数真的
    到 20+，可以考虑用字典分发表，但现在不必要。"

2. 指出关键难点
   "PLACE 这种有参数的命令比较特殊，分发表只适合无参命令，
    所以可能要分两部分处理。"

3. 强调权衡
   "工厂模式、策略模式都能解决，但对这个题太复杂。
    我选最轻量的方案。"
```

❌ **要避免的做法**：
- 一开始就提"工厂模式""策略模式"（显得爱过度设计）
- "代码可以重构"但完全不给具体方案
- 固守初始方案，不愿讨论改进

## 关键评估维度

### 1. Technical Knowledge 的信号

**强信号（✅）**：
- 快速阅读并理解现有代码
- 能识别复用的机会（不重复代码）
- 知道边界检查的必要性
- 能说出数据结构选择的理由
- 考虑状态一致性和原子性

**弱信号（❌）**：
- 边界情况考虑不周（机器人在边缘行为）
- 代码逻辑重复
- 对 Bug 反应慢（写完不验证）
- 无法解释为什么选择某个数据结构

### 2. Pragmatism 的信号

**强信号（✅）**：
- 先实现最小可工作版本
- "这样够用，如果需要再优化"
- 时间分配合理（不纠缠细节）
- 能做出权衡决策并说出理由
- 接受"不完美"的中间方案

**弱信号（❌）**：
- 一开始就设计复杂架构
- 花 20 分钟在变量命名或注释
- 对临时方案感到不安
- 试图做得完美而压低速度

### 3. Learning Mindset 的信号

**强信号（✅）**：
- 主动提问澄清需求
- "这是个好建议，我改"
- 能解释自己的思路和权衡
- 愿意讨论多个方案
- 坦承不确定的地方

**弱信号（❌）**：
- 沉默编码，不说想法
- 被指出问题后防守或反驳
- "我选这个因为……"但说不出好理由
- 不提问，硬猜需求

## 回答策略

### 通用框架

对**任何**面试官的问题，使用这个结构：

```
【第一步】确认需求
"我理解对吗？你要……"
"在实现之前，我想确认一下……"

【第二步】说出至少两个选项（及权衡）
"我可以选择 A（优点/缺点）或 B（优点/缺点）"

【第三步】做出决策并说出理由
"我选 A，主要因为……"

【第四步】实现并验证
[写代码，跑测试]

【第五步】对反馈开放
"我一开始没想到那个问题，谢谢指点。
 那这样改是否更好……？"
```

### 处理"不知道"的情况

**错误做法**：
- ❌ "我不知道"（显得被动）
- ❌ 装作知道（面试官会追问，露馅更尴尬）
- ❌ "那个超出我的范围"（显得缺乏好奇）

**正确做法**：
- ✅ "我没遇过这个，但我可以这样思考……"
- ✅ "我倾向选 X，但我想听听你的想法"
- ✅ "这个我需要查一下文档/再想想"

---

### 处理被指出错误

**场景**：
```
面试官："嗯，但这样有个问题……当机器人在边缘时……"
```

**你应该说**：
```
✅ "哦，我没想到！那……"
✅ "这是个好点子。让我改成……"
✅ "我同意，这样更安全。"

❌ "不会的，因为……" （显得不愿接纳意见）
❌ "那是边界情况，可以以后再处理" （显得考虑不周）
❌ 继续坚守原方案 （显得固执）
```

## 常见追问

### 关于状态管理

| 面试官问 | 你应该说 | 为什么这样说 |
|---------|---------|-----------|
| "UNDO 存储什么？" | "我存 (x, y, facing, move_count) 四元组" | 完整且明确 |
| "如果 UNDO 多次会怎样？" | "历史栈空了就返回失败，这是合理的" | 显示考虑了边界 |
| "move_count 在 UNDO 后是否变化？" | "应该恢复到之前的值，因为我们完全回到那个状态" | 强调原子性 |

### 关于碰撞检查

| 面试官问 | 你应该说 | 为什么这样说 |
|---------|---------|-----------|
| "碰撞检查应该在哪儿？" | "在 Table，因为 Table 是真理来源" | 职责清晰 |
| "两个机器人同时 MOVE 到一个格子？" | "命令序列执行，先到的就占了，后来的会失败" | 简单且实用 |
| "A 推 B，B 推不动怎么办？" | "整个推动链失败，A 回到原位" | 强调原子性 |

### 关于扩展性

| 面试官问 | 你应该说 | 为什么这样说 |
|---------|---------|-----------|
| "以后加 20 个命令？" | "现在 if-elif 够用，如果真到 20+ 可以用分发表" | 展示成熟的判断 |
| "支持 100 个机器人？" | "二维网格变成瓶颈，可以改成稀疏表示" | 有扩展意识 |
| "支持 undo 重放？" | "需要分别记录命令历史和状态历史，不能混淆" | 理解根本差异 |

## 红旗与绿旗

### 🚩 Red Flags（面试官看到会减分）

1. **沉默长达 5+ 分钟**
   - 无法展示思路
   - 面试官认为你可能卡住了

2. **代码写完不验证**
   - "应该能用"（但没跑过）
   - 问"能帮我跑一下吗？"

3. **边界情况完全遗漏**
   - BACKWARD 在角落会怎样？未考虑
   - UNDO 到底了怎么办？未考虑

4. **被指出问题后防守**
   - 坚守错误的方案
   - 给理由为什么面试官的想法不对

5. **过度设计初期方案**
   - 加 BACKWARD 就上设计模式
   - "为了以后可扩展"实现过度

6. **不提问，硬猜**
   - 既没问"你们想要什么"
   - 也没问"我的理解对吗"

### 🟢 Green Flags（面试官看到会加分）

1. **快速理解代码**
   - 5 分钟内读懂现有实现
   - 能说出关键数据结构（如 DIRECTION_DELTAS）

2. **主动问澄清问题**
   - "在实现之前我想确认……"
   - "你希望我们优先做哪个方向？"

3. **说出权衡并做决策**
   - "我可以选 A 或 B，我倾向 A 因为……"
   - 有理有据，不是随意

4. **增量开发和验证**
   - "写完了，我跑一下"
   - 每个小步骤都验证

5. **接纳反馈并调整**
   - "这是个好建议"
   - "我改成这样是否更好？"

6. **诚实暴露局限**
   - "目前没处理 X，如果需要我可以……"
   - 不装全知道

7. **展现学习心态**
   - 主动问"怎么优化？"
   - "我之前没想到那个，谢谢"

## 面试官通常不会问的

### ❌ 算法竞赛问题
- LeetCode hard 难度的算法题
- 数学推导或复杂数学公式
- 脑筋急转弯

### ❌ 深度语言特性
- Python 装饰器的细节（虽然会用就行）
- 并发、多线程、异步的深度知识
- 垃圾回收机制

### ❌ 系统设计
- 分布式系统架构
- 数据库分片策略
- 完整的微服务设计（除非明确问"设计支持 1000 个机器人"）

## 你应该准备的问题（提问权利）

面试最后 5-10 分钟，面试官会问"你有什么问题吗？"。这是你反过来评估团队和职位的机会，也是展示你专业度的时刻。

### 提问的目的

✅ **好的目的**：
- 了解真实的工作环境和团队文化
- 展示你对职位的认真态度
- 了解职业发展机会
- 确认自己是否适合这个岗位

❌ **不好的目的**：
- 只想了解薪资和休假
- 显得你没有准备
- 完全不提问（显得不关心）

---

### 推荐提问（分类）

#### 1. 关于团队协作

```
"从你的角度看，这个岗位需要最强的技能是什么？"
```
**为什么好**：了解招聘方最看重什么，也能知道团队的痛点

```
"Voyager 团队日常用 pair programming 吗？具体怎么操作的？"
```
**为什么好**：确认面试中谈到的 pair programming 是否真的在用

```
"团队的代码审查流程是怎样的？通常需要多少人 approve？"
```
**为什么好**：了解代码质量标准和协作模式

#### 2. 关于项目和挑战

```
"现在团队面临的最大技术挑战是什么？"
```
**为什么好**：了解是否有技术成长的机会，团队是否务实

```
"这个职位的前 30 天/90 天重点交付是什么？"
```
**为什么好**：理解期望，知道如何开始

```
"如果我加入，会主要参与哪些项目或功能？"
```
**为什么好**：具体了解工作内容

#### 3. 关于成长和 Onboarding

```
"新入职的人通常多久能独立交付代码？"
```
**为什么好**：了解学习曲线和支持体系

```
"有导师制或 onboarding buddy 吗？"
```
**为什么好**：了解是否有清晰的成长路径

```
"团队主要用什么技术栈？有没有定期的技术分享？"
```
**为什么好**：了解学习环境

#### 4. 关于工作文化

```
"远程工作政策是怎样的？多久需要在办公室？"
```
**为什么好**：了解实际工作安排

```
"工作时间的灵活性怎样？是固定时间还是 flexible？"
```
**为什么好**：理解工作节奏

```
"团队最喜欢的工作特点是什么？"
```
**为什么好**：了解团队的价值观

### 要避免的提问

**❌ 太功利**
- "你们加班多吗？"
- "假期有多少天？"
- "年终奖怎么算？"

➡️ **原因**：现在不是讨论这些的时候，属于 HR 轮

**❌ 属于 HR 的话题**
- "薪资范围是多少？"
- "签约年限是几年？"
- "是否有期权？"

➡️ **原因**：显得你只在乎钱，而且 HR 会更专业地回答

**❌ 显得没准备**
- "这个职位是做什么的？"（你应该读过 JD）
- "公司是做什么的？"（你应该做过调研）
- 完全没有问题

➡️ **原因**：显得你没有认真准备

**❌ 过于挑战性**
- "为什么我看你们代码有这么多 bug？"
- "为什么离职率这么高？"

➡️ **原因**：显得你在评判，不够尊重

### 提问的技巧

#### 💡 1. 基于面试内容提问

**好的方式**：
```
面试官："我们最近在重构 UNDO 逻辑"
你："你刚才提到重构 UNDO，能告诉我那个挑战主要在哪儿吗？"
```

**为什么好**：显示你在认真听，问题有针对性

#### 💡 2. 问 2-3 个问题足够

不要问 5 个以上的问题。质量比数量重要。

**好的节奏**：
```
1. 问一个关于团队/项目的问题
2. 问一个关于成长的问题
3. (可选) 问一个关于工作文化的问题
```

#### 💡 3. 根据面试官的回应追问

```
面试官："我们用 Python 和 FastAPI"
你："团队在 FastAPI 上有多久了？有人特别精通吗？"
```

#### 💡 4. 准备笔记本

在面试前把你的问题写下来，这样：
- 不会忘记
- 显得专业
- 如果想不起来可以翻阅

#### 💡 5. 根据你的表现调整

- **如果面试很顺利**：你可以问更深度的问题，比如"团队的技术决策过程怎样？"
- **如果面试有点卡顿**：问一些更温暖的问题，比如"团队最喜欢合作的地方是什么？"

## 最后检查清单

面试前：
- [ ] 读完 `instructions.md` 和 `pair-programming-prep.md`
- [ ] 跑一遍现有代码的测试，确保环境 OK
- [ ] 自己实现一遍 BACKWARD 和 UNDO，熟悉手感
- [ ] 测试 Zoom 屏幕共享，代码清晰度

面试中：
- [ ] 破冰 5 分钟，简短自我介绍和了解面试官
- [ ] 理解需求 10 分钟，克隆、读代码、跑测试、确认任务
- [ ] 编码时**不要沉默超过 2 分钟**，一定要说出想法
- [ ] 每完成一个小功能立即运行验证
- [ ] 被指出问题时**立即接纳**，不要防守

面试后：
- [ ] 演示最终成果
- [ ] 说明时间不够做的地方
- [ ] 提 2-3 个好问题
- [ ] 感谢面试官

## 如何从题目判断难度级别

**初始问题听到这些词 → Level 1**：
- "添加"、"扩展"、"新命令"
- "BACKWARD"、"JUMP"
- 时间提示 "15 分钟"

**初始问题听到这些词 → Level 2**：
- "历史"、"UNDO"、"状态"
- "时间倒流"
- 时间提示 "20 分钟"

**初始问题听到这些词 → Level 3**：
- "多个"、"碰撞"、"障碍"、"同时"
- 时间提示 "30 分钟" 或"如果时间够"

**初始问题听到这些词 → Level 4**：
- "文件"、"架构"、"设计"、"重构"
- "如果……以后……怎么办"

## 总结

REA 的 pair programming 面试，根本不是在考"你能有多快独立完成任务"，而是在看：

1. **你能否和我们一起思考问题** — 这是 Technical Knowledge
2. **你能否用简单方案解决复杂问题** — 这是 Pragmatism  
3. **你能否接纳不同的想法和协作** — 这是 Learning Mindset

所以：
- ✅ **主动沟通** — 面试官看不到你脑子里的想法，一定要说出来
- ✅ **增量验证** — 不要写完 100 行再跑，边写边测
- ✅ **坦诚和谦虚** — 不确定就问，出错就改，这才是专业
- ✅ **克制设计** — 用最简单的方式解决，不是最聪明的方式

Good luck! 记住：这不是考试，是和未来同事一起写代码。
