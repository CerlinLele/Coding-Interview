## 技术错误

### 错误 1：边界情况遗漏

#### 场景

```python
def backward(self):
    dx, dy = self.DIRECTION_DELTAS[self.facing]
    new_x = self.x - dx
    new_y = self.y - dy
    self.x = new_x
    self.y = new_y
    return True
```

**问题**：没有检查新位置是否合法（可能越界）

#### 面试官会怎样测试

```
Q: "机器人在 (0,0) 面向南，BACKWARD 会怎样？"
A: [代码让机器人飘到 (-1, 0)，越界了]
```

#### 正确做法

```python
def backward(self):
    dx, dy = self.DIRECTION_DELTAS[self.facing]
    new_x = self.x - dx
    new_y = self.y - dy
    
    # ✅ 必须检查
    if self.table.is_valid_position(new_x, new_y):
        self.x = new_x
        self.y = new_y
        return True
    
    return False
```

#### 预防方法

- 写完功能立即运行边界测试
- 自己问自己："如果在角落会怎样？"
- 每个涉及坐标的操作都要想到 `is_valid_position()`

### 错误 2：状态不一致（UNDO 的噩梦）

#### 场景

```python
def move(self):
    self.x = self.x + dx
    self.y = self.y + dy
    # ❌ 忘了更新 Table 的网格
    # ❌ 忘了更新 move_count
    self.history.append((self.x, self.y, self.facing))
    return True
```

**问题**：Robot 自己的位置更新了，但：
- Table 的二维网格没同步 → 碰撞检查失效
- move_count 没增加 → REPORT 信息错误
- 后续 UNDO 恢复位置但网格还是旧的 → 死锁

#### 面试官会怎样发现

```
面试官："现在两个机器人在同一格子里？"
你："哦……我的 move() 没有更新 Table 的网格"
```

#### 正确做法

```python
def move(self, direction="forward"):
    if not self.is_placed():
        return False
    
    dx, dy = self.DIRECTION_DELTAS[self.facing]
    new_x = self.x + dx
    new_y = self.y + dy
    
    # 1️⃣ 检查合法性
    if not self.table.is_valid_position(new_x, new_y):
        return False
    
    # 2️⃣ 保存旧位置用于 UNDO
    self.history.append((self.x, self.y, self.facing))
    
    # 3️⃣ 清除旧位置在网格上的占用（如果有）
    self.table.robots[self.x][self.y] = None
    
    # 4️⃣ 更新新位置
    self.x = new_x
    self.y = new_y
    self.table.robots[self.x][self.y] = self
    
    # 5️⃣ 更新计数（只有移动才算，转向不算）
    self.move_count += 1
    
    return True
```

#### 预防方法

- 想象有个"一致性检查员"在看，每个改动都要同步到所有相关数据
- 写完立即手动验证："robot.x 是多少？table.robots[x][y] 是什么？"
- 多机器人测试时立即发现

### 错误 3：复用不当导致 Bug 链

#### 场景

```python
def left(self):
    self.facing = (self.facing - 1) % 4
    self.history.append((self.x, self.y, self.facing))  # ❌ 不该存

def right(self):
    self.facing = (self.facing + 1) % 4
    self.history.append((self.x, self.y, self.facing))  # ❌ 不该存
```

**问题**：LEFT/RIGHT 旋转不动，不应该记录到 UNDO 历史

#### 面试官会怎样发现

```
面试官："UNDO 一个 LEFT 命令，会回到哪儿？"
你："……我把所有命令都记了……"
面试官："那 UNDO 5 次是什么意思？"
你："[意识到逻辑错了]"
```

#### 正确做法

```python
def move(self, direction="forward"):
    # 只有这个方法保存历史
    self.history.append((self.x, self.y, self.facing))
    # ... 移动逻辑

def left(self):
    # ✅ 不保存历史，因为没移动
    self.facing = (self.facing - 1) % 4
    return True

def right(self):
    # ✅ 不保存历史
    self.facing = (self.facing + 1) % 4
    return True

def undo(self):
    # UNDO 恢复到上一个有效的 "状态"
    # 是指最后一次 move，不是最后一条命令
    if not self.history:
        return False
    self.x, self.y, self.facing = self.history.pop()
    return True
```

#### 预防方法

- 问自己："这个操作改变了位置吗？"
  - 改变 → 记录
  - 不改变 → 不记录
- 写完测试："LEFT 后 UNDO，位置应该不变"

### 错误 4：参数解析脆弱

#### 场景

```python
def execute(self, command):
    if command.startswith('PLACE'):
        parts = command.split(',')
        x = int(parts[0])  # ❌ IndexError 如果输入不完整
        y = int(parts[1])
        facing = parts[2]
```

**问题**：对非法输入没有防御

#### 面试官会怎样测试

```
"试试 PLACE 1,2（缺少朝向）？"
[RuntimeError: list index out of range]
```

#### 正确做法

```python
def execute(self, command):
    command = command.strip().upper()
    
    if command.startswith('PLACE'):
        parts = command.split()  # 用空格分割更清晰
        if len(parts) != 4:  # PLACE X Y FACING
            return {"success": False, "message": "Invalid PLACE format"}
        
        try:
            x = int(parts[1])
            y = int(parts[2])
            facing = parts[3]
        except (ValueError, IndexError) as e:
            return {"success": False, "message": f"Invalid PLACE arguments: {e}"}
        
        return self.place(x, y, facing)
    
    # ... 其他命令
```

#### 预防方法

- 先想想非法输入有哪些：缺少参数、参数不是数字、拼写错误
- 用 try-except 或提前检查长度
- 有好的错误消息比 silent fail 好得多

## 沟通错误

### 错误 5：沉默敲代码超过 2 分钟

#### 场景

```
[你在屏幕前安静地打字 5 分钟]
面试官：[看不到你在想什么，开始紧张]
```

**问题**：面试官无法了解你的思路，以为你卡住了

#### 面试官的心理

```
"他/她在想什么？是不是卡住了？
 应该问一下吗？但打断他/她不好……
 这个表现有点可疑……"
```

#### 正确做法

```
你："让我看一下现有的 move() 实现……"
   [读代码，边读边说]
   "好的，它用了 DIRECTION_DELTAS 字典……"
   [停顿，思考]
   "BACKWARD 就是反向移动。我可以参数化 move 方向，
    或者写个新方法。我倾向参数化因为……"
   [开始打字]
   "现在写 move(direction='forward') 的签名……"
```

#### 预防方法

- **设定2分钟闹钟**：如果超过 2 分钟没说话，停下来说点什么
- 说出你的想法，哪怕只是"让我想想"
- 边写边说，即使内容很简单

### 错误 6：被指出问题后防守

#### 场景

```
面试官："嗯，这里有个问题。如果历史栈为空呢？"
你："哦……其实这种情况很罕见……"
    或
    "但这样已经够用了……"
```

**问题**：显得固执，不愿接纳意见

#### 面试官的感受

```
"他/她在为自己辩护而不是听意见。
 这样的人在 code review 时会很难合作。"
```

#### 正确做法

```
面试官："嗯，这里有个问题。如果历史栈为空呢？"
你："哦！我没想到这个。那我应该加上栈为空的检查……"
   [快速修改]
   "这样是否好多了？"
   
或者，如果你本来想过这个：
你："对，我刚才其实想到了。我倾向返回 False，
    因为没什么可 UNDO 的。这样行吗？"
```

#### 预防方法

- 把面试官的反馈看作**礼物而不是批评**
- 快速调整，而不是解释为什么你的方案其实没问题
- 说"谢谢，这是个好点子"，然后改

### 错误 7：过度补充，显得不自信

#### 场景

```
你写完 BACKWARD，说："这就是 BACKWARD 的实现。
其实还有其他方式……可以用 state pattern……
或者 command pattern……
或者……"
```

**问题**：显得对自己的选择不有信心

#### 面试官的感受

```
"他/她似乎在试图用复杂的东西给我留下印象，
 但其实显得不自信。"
```

#### 正确做法

```
你："这就是 BACKWARD 的实现。我选了参数化方案
    因为简单且易于维护。如果有什么可以改进的，
    我很乐意听。"

或者，如果面试官问：
面试官："还有其他方式吗？"
你："当然。如果命令数量很多，可以考虑策略模式或
    分发表。但现在 if-elif 已经很清晰了。"
```

#### 预防方法

- 陈述你的选择，然后**停下来**
- 等面试官问"还有其他方式吗？"再讨论替代方案
- 自信但不傲慢

## 流程错误

### 错误 8：跳过验证，一直往前写

#### 场景

```
15 分钟过去，你写了 BACKWARD、JUMP、UNDO，
但都没运行过。
面试官："能跑一下吗？"
[一堆 bug 出现]
```

**问题**：错误堆积，难以调试，浪费时间

#### 面试官的感受

```
"他/她写代码很快，但没有工程纪律。
 在生产环境这样会出大问题。"
```

#### 正确做法

```
步骤 1：写 BACKWARD 骨架（2 分钟）
步骤 2：立即跑一下试试（1 分钟）
步骤 3：边界测试（1 分钟）
步骤 4：重构/优化（可选，1 分钟）
---
步骤 5：写 JUMP 骨架
步骤 6：立即验证
... 以此类推
```

#### 预防方法

- **每 5-10 分钟运行一次**
- 边界测试不必很全，但边界必须想到
- 运行失败是**信息，不是失败**

---

### 错误 9：没有澄清需求就开始编码

#### 场景

```
面试官："实现 UNDO"
你：[立即开始写]
```

30 分钟后……
```
你："写完了"
面试官："哦……我想要的是全局 UNDO，
         而不是单个机器人的 UNDO"
你："哦……那得重写……"
```

**问题**：时间浪费，需求理解偏差

#### 面试官的感受

```
"他/她不确认需求就开始。这样在实际项目中
 容易做出不是产品经理想要的东西。"
```

#### 正确做法

```
面试官："实现 UNDO"
你："让我确认一下。UNDO 是撤销最后一个 MOVE 吗？
     还是撤销任何操作（包括 LEFT/RIGHT）？"
面试官："只撤销 MOVE，不撤销转向。"
你："好的，还有一个问题。UNDO 多次会怎样？
     回到未放置状态吗？"
面试官："是的，可以。"
你："明白了，我现在开始……"
```

#### 预防方法

- **先问后做**，多花 1-2 分钟澄清
- 关键词：
  - "我理解对吗……?"
  - "还有需要特别处理的吗？"
  - "边界情况怎么处理？"

---

### 错误 10：时间分配不当

#### 场景

```
【60 分钟后】
你才刚完成 Level 1 的 BACKWARD，还没有完全测试
还想继续优化……

面试官："我们还有 30 分钟，要不要试试 Level 2？"
你："不……我想把这个做完美……"
```

**问题**：追求完美导致后续无法进行

#### 面试官的感受

```
"他/她没有全局的时间意识。
 做到 80 分就应该往前走，不是一直打磨 100 分。"
```

#### 正确做法

```
【第 10 分钟】完成 BACKWARD 骨架 + 基础测试
【第 15 分钟】被问到边界和优化，快速处理
【第 20 分钟】说"BACKWARD 可以了，我们往前进吧"

面试官："很好。现在要不要试试 UNDO？"
你："好的！"
```

#### 预防方法

- 定心锚："完成 → 验证 → 下一个"
- 如果面试官没追问，就别继续优化
- 优先广度（做多个功能），而不是深度（完美一个功能）

## 心态错误

### 错误 11：一开始就被紧张打垮

#### 场景

```
"嗨，准备好了吗？我们开始吧。"
你：[脑子一片空白，开始快速敲代码，
      完全没思路，最后一团糟]
```

**问题**：没有缓冲，被紧张绑架

#### 正确做法

```
面试官："嗨，准备好了吗？我们开始吧。"
你："好的！让我先读一下代码库……"
   [深呼吸，花 2 分钟看一眼 README]
   "好的，现在我理解了……"
   [再花 1 分钟理清思路]
   "我现在开始……"
```

#### 预防方法

- 提前 5 分钟进会议，测试设备和心态
- 读代码的前 5 分钟是"缓冲"，不是"开始"
- 一开始慢速理解，往往整体更快

---

### 错误 12：完美主义心态

#### 场景

```
"我必须把所有边界都处理完美，
 否则面试官会认为我不专业……"

[结果花 30 分钟写 100 行高度防御的代码，
 完成度不到 50%]
```

**问题**：追求完美导致无法交付

#### 面试官的感受

```
"他/她的代码很严谨，但只做了半个功能。
 在真实项目中，宁愿 80% 完成度 5 个功能，
 也不要 100% 完成度 1 个功能。"
```

#### 正确做法

```
你的心态：
"我先做一个 MVP（最小可工作版本）。
 时间允许的话再加防御性的代码。"

实际行动：
【10 分钟】完成最小实现
【5 分钟】跑基本测试
【5 分钟】补充边界处理
【5 分钟】可选的优化
```

#### 预防方法

- 把 80/20 法则刻在脑子里
- "完成 > 完美"
- 问面试官："还有其他方向我应该覆盖吗？"

### 错误 13：被之前的代码吓倒

#### 场景

```
你读现有代码，发现用了一些你不熟悉的特性
（比如 deque、property、__dunder__ 方法），
于是完全丧失信心……

"我可能不适合这个职位……"
```

**问题**：过度关注工具细节，忽视问题解决能力

#### 面试官的感受

```
"他/她理解不了现有代码，可能 level 比较初级。"
```

#### 正确做法

```
你读到陌生代码时：
"我不太熟悉这个写法，但我可以……"

快速查证：
"让我搜一下 deque 是什么……"
或
"这看起来像是一个双端队列，对吧？"

然后继续：
"好的，现在我理解了。我可以继续了。"
```

#### 预防方法

- 承认不知道，而不是装知道或放弃
- "我对 X 不太熟悉，但我可以快速学"
- 这恰好展示了 Learning Mindset

### 错误 14：过度解释自己的选择

#### 场景

```
你完成了 BACKWARD，面试官还没问，
你就开始：

"我选择参数化而不是写新方法，因为……
 DRY 原则……信息论……
 还有伯克利计算机科学系的研究表明……"
```

**问题**：显得啰嗦，浪费时间，显得不自信

#### 面试官的感受

```
"他/她在试图用解释来说服我他/她的选择是对的。
 这显得他/她对自己的决定没信心。"
```

#### 正确做法

```
你："写完了。我选了参数化 move()，
    理由是减少重复代码。"
   [停下来，等面试官反馈]

如果面试官问：
面试官："为什么不写新方法？"
你："那样就复制了逻辑，以后改 boundary check 
    要改两处。参数化方法一处改就行。"
```

#### 预防方法

- 3 句话原则：说出你的选择、理由、然后停下来
- 除非面试官追问，否则不要扩展解释

## 快速自检清单

在面试前 1 小时，问自己：

### 技术准备

- [ ] 我能在 5 分钟内理解 toy robot 代码结构吗？
- [ ] 我知道 `DIRECTION_DELTAS` 的作用吗？
- [ ] 我能说清楚 `is_valid_position()` 为什么重要吗？
- [ ] 我想过 BACKWARD 在边缘会怎样吗？
- [ ] 我知道 UNDO 应该存什么状态吗？

### 沟通准备

- [ ] 我能听清面试官的问题吗？（网络清晰？声音清晰？）
- [ ] 我准备好了说"让我确认一下……"吗？
- [ ] 我知道不能沉默超过 2 分钟吗？
- [ ] 我准备好被指出错误后说"谢谢"而不是"但是"吗？

### 流程准备

- [ ] 我有时间意识吗？（知道什么时候该往前走）
- [ ] 我知道每 5-10 分钟应该运行一下吗？
- [ ] 我知道 MVP 原则吗？（80% 完成度比 100% 更好）
- [ ] 我准备好问"还有其他需要吗？"吗？

### 心态准备

- [ ] 我相信自己能完成 Level 1 吗？
- [ ] 我对 Level 2-3 有信心吗？（即使做不完也不会慌张）
- [ ] 我知道完美性不是成功标准吗？
- [ ] 我准备好学习而不是证明自己吗？

如果以上大部分都打勾，你已经准备好了！

## 最后的话

这些错误中，**最容易挽救的是沟通错误**。

一个有 bug 但能清晰解释思路、能接纳意见、能快速调整的候选人，往往比一个完美但沉默、固执的候选人更受欢迎。

所以如果你只能改一件事，那就是：**保持开放的沟通**。

祝你好运！
