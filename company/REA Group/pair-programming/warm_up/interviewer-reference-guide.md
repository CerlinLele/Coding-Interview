# REA Group 面试官快速参考表

这份文档是为面试官设计的，用来快速参考在不同阶段应该问什么问题。

---

## 破冰阶段（5 分钟）

### 目标
- 让候选人放松
- 了解候选人的背景
- 建立协作氛围

### 标准开场

```
"嗨 [名字]，欢迎！我是 [名字]，在 REA 做 [团队]。
我的同事 [名字] 也在这。我们今天来一起做个代码练习。
你在这个领域有多少年经验？用过 Python 吗？"
```

### 观察点
- ✅ 候选人是否放松和主动介绍背景
- ❌ 过于紧张或完全不说话

---

## 理解需求阶段（10 分钟）

### 你的任务

给候选人 toy robot 代码库的 GitHub 链接，让他们：
1. 克隆到本地
2. 读 README
3. 跑现有测试
4. 总结他们的理解

### 标准引导

```
"代码库我已经分享给你了。你可以克隆到本地，
用熟悉的编辑器打开。把现有的测试跑一遍，
然后告诉我你对当前代码的理解。"
```

### 观察点

| 观察项 | 强信号 ✅ | 弱信号 ❌ |
|--------|---------|---------|
| **理解速度** | 5-8 分钟读懂主要结构 | 15+ 分钟还在摸索 |
| **验证** | 主动跑测试 | 跳过测试直接开始 |
| **总结** | 能说出关键类和数据结构 | "嗯……就是一个机器人……" |
| **提问** | 主动澄清需求 | 完全不提问 |

### 标准确认问题

```
"所以现在的代码支持 [他们的总结]。
我们接下来要加 [功能]，听起来合理吗？"
```

## Level 1 编码阶段（10-15 分钟）

### 初始题目（选一个）

**选项 A - 简单扩展**：
```
"能不能给机器人加一个 BACKWARD 命令？
 功能就是向后退一步。"
```

**选项 B - 坐标跳转**：
```
"能不能实现 JUMP X,Y？
 让机器人直接跳到目标位置，保持朝向。
 如果目标位置越界或有障碍就跳不了。"
```

### 时间检查点

- **3 分钟**：候选人应该开始写代码
- **7-10 分钟**：功能应该大致完成并运行
- **12 分钟**：应该跑过测试或手动验证

### 第一轮追问（如果代码完成）

```
"很不错。我看你 BACKWARD 和 MOVE 有点重复。
 能优化一下吗？"
```

**观察**：
- ✅ 立即看到问题，解释 DRY 原则
- ✅ "对，我可以提取共同部分……"
- ❌ "这样已经够用了"（不愿改进）
- ❌ 开始过度设计

### 第二轮追问（如果做了优化）

```
"如果以后要加 DIAGONAL（斜向移动），
 现在的参数设计还够用吗？"
```

**观察**：
- ✅ "现在是字符串参数，如果太多可以考虑 Enum，但现在不需要"
- ✅ 展现了可扩展性意识但保持务实
- ❌ "我早就用工厂模式了……"（过度设计）
- ❌ "现在不管那么多"（完全不考虑）

### 边界情况提问（如果时间充足）

```
"机器人在 (0,0) 面向北，BACKWARD 会怎样？
 你有测过这个吗？"
```

**观察**：
- ✅ "我测过了，会被阻挡在原位，返回失败"（验证过）
- ✅ "应该会被挡住，让我现在跑一下看看"（快速验证）
- ❌ "应该能……"但没验证过
- ❌ "哦，我没想到这个"（不够全面）

## Level 2 编码阶段（15-20 分钟）

### 初始题目

```
"现在要加 UNDO 功能。怎么设计？"
```

### 第一步：听他们的方案

**好的开场**：
```
你："我会在 Robot 里加一个 history 栈，
    保存每次操作前的状态 (x, y, facing)。
    UNDO 时就弹出并恢复。"
```

**观察**：
- ✅ 说出了状态快照方案和对应的权衡
- ✅ 提到了数据结构（栈）
- ❌ "嗯……我还在想"（完全没想法）
- ❌ "我用复杂的事件系统……"（过度）

### 第二步：深入追问

如果候选人说用状态快照：
```
"快照里应该存什么？只有位置吗？
 如果有 move_count，UNDO 后是否应该减少？"
```

**观察**：
- ✅ "应该存 (x, y, facing, move_count)，因为我们要完全回到那个状态"
- ✅ 意识到需要同步 move_count
- ❌ "只存位置就够了"（不完整）
- ❌ "UNDO 后 move_count 不变"（逻辑不一致）

### 第三步：边界情况

```
"连续 UNDO 多次会怎样？
 历史栈到底了呢？"
```

**观察**：
- ✅ "历史栈空了就返回失败，这是合理的行为"
- ✅ 能解释"无处可退"的情况
- ❌ "没想过这个"（考虑不周）
- ❌ "我可以用 Optional……"（引入不必要的复杂性）

### 第四步：扩展追问（如果时间充足）

```
"现在要加 REDO 功能。怎么处理？
 关键是什么？"
```

**观察**：
- ✅ "需要第二个栈。关键是任何新的 MOVE/LEFT/RIGHT 都要清空 redo 栈"
- ✅ 识别了"进入新时间线"的隐含规则
- ❌ "不知道……"（缺乏基础知识）
- ❌ "用单个栈可以记录双向"（不对）

## Level 3 编码阶段（20-30 分钟）

### 初始题目

```
"现在要支持多个机器人在同一个桌子上。
 机器人之间不能碰撞，也不能进入障碍物。
 怎么改架构？"
```

### 第一步：理解方案

```
"在开始编码之前，给我讲讲你的思路。
 Table 应该怎么存储多个机器人？"
```

**好的回答框架**：
```
你："我会让 Table 维护一个机器人字典或二维网格。
    每次机器人 MOVE 前，调用 Table.is_valid_position()
    来检查有没有其他机器人或障碍。"
```

**观察**：
- ✅ 清楚的职责划分（Table 存数据，Robot 请求验证）
- ✅ 提到了 API 设计（is_valid_position 增强）
- ❌ "我不知道该怎么改"（需要指导）
- ❌ "我会用消息队列……"（远离问题域）

### 第二步：API 设计追问

```
"怎么初始化多个机器人？命令格式怎么改？"
```

**好的回答**：
```
你："我可能需要给机器人加个 name 或 id。
    命令可能是 'PLACE 1,1,NORTH ROBOT1'
    或者 'R1 PLACE 1,1,NORTH'。
    两种都有利有弊，你们怎么想？"
```

**观察**：
- ✅ 主动提出多个方案
- ✅ 邀请面试官反馈
- ❌ 假定一个方案而不确认
- ❌ 完全无法扩展现有 API

### 第三步：碰撞检查

```
"碰撞检查应该在哪儿实现？
 Robot 里还是 Table 里？"
```

**好的回答**：
```
你："Table。因为 Table 是唯一知道整个占用状态的。
    Robot 只管申请移动，由 Table 决定是否合法。"
```

**观察**：
- ✅ 职责清晰，有一致的哲学
- ✅ 能解释为什么这样划分
- ❌ "可能都可以"（摇摆不定）
- ❌ 把碰撞逻辑分散在两个地方

### 第四步：复杂场景（如果时间充足）

```
"如果机器人 A 要推动机器人 B，怎么处理？
 B 推不动呢？整个操作失败吗？"
```

**好的回答**：
```
你："我觉得应该原子化。A 推 B 前先检查 B 是否能移动。
    如果 B 无处可去，整个推动链失败，A 保持原位。
    这样更安全。"
```

**观察**：
- ✅ 考虑了原子性和一致性
- ✅ 能解释设计选择
- ❌ "推一半，推不动就停"（状态不一致）
- ❌ "我没想过这个"（没有深度思考）

## Level 4 讨论阶段（如果时间充足）

### 题目类型

这一阶段通常不是"实现什么"，而是"如果……怎么设计"。

### 追问示例

#### 何时抽象

```
"现在有 7 个命令用 if-elif。
 如果以后到 50 个怎么办？"
```

**强回答**：
```
"现在就上复杂设计不值得。我会看命令数真的上升，
 然后考虑用字典分发表。策略模式太重了。"
```

**弱回答**：
```
"我从一开始就设计工厂模式……"
或
"将来再说吧"
```

#### 扩展性

```
"支持 100×100 的桌子和 50 个机器人。
 现有的二维网格设计会有问题吗？"
```

**强回答**：
```
"内存占用会很大。可以考虑稀疏表示—只存有机器人和障碍物的格子。"
```

**弱回答**：
```
"应该没问题……"
或
"我不知道……"
```

#### 持久化

```
"想把游戏状态保存到文件。怎么序列化？"
```

**强回答**：
```
"用 JSON。存位置、朝向、障碍物位置。
 uuid 要 str() 转换。如果要保留 undo 能力
 可能还要存 history，但现在我先不存。"
```

**弱回答**：
```
"用 pickle"
或
"没想过……"
```

## 综合评分表

在编码结束后，用这个表格快速评分：

### Technical Knowledge（技术基础）

| 分数 | 表现 |
|-----|-----|
| 5 | 快速理解现有代码；能识别复用机会；考虑边界情况；能说清设计选择 |
| 4 | 较快理解代码；大多数边界情况考虑；能解释设计选择 |
| 3 | 理解有延迟；遗漏一些边界情况；设计选择说不清 |
| 2 | 理解困难；重大边界遗漏；代码逻辑重复 |
| 1 | 基本无法理解现有代码；频繁 bug；完全无设计思考 |

### Pragmatism（务实性）

| 分数 | 表现 |
|-----|-----|
| 5 | 快速做决策；优先最小可工作版本；能做权衡并说清理由；时间分配合理 |
| 4 | 决策比较快；能优先完成核心功能；知道权衡 |
| 3 | 决策有延迟；有时陷入细节；部分权衡考虑 |
| 2 | 花时间在不重要的事；完全不考虑权衡；追求完美 |
| 1 | 一开始就过度设计；时间管理很差；无法交付 |

### Learning Mindset（学习态度）

| 分数 | 表现 |
|-----|-----|
| 5 | 主动提问；立即接纳反馈；主动讨论多个方案；坦诚不确定；好奇心强 |
| 4 | 偶尔提问；能接纳反馈；讨论方案 |
| 3 | 少提问；被指出错误后能改；有些固执 |
| 2 | 完全不提问；对反馈有防守；难以沟通 |
| 1 | 沉默寡言；被指出错误后坚守；显得不愿学习 |

### 整体评价

```
【Pass】
- 三个维度都 ≥ 3
- 或者任一维度 5，其余 ≥ 3

【Strong Pass】
- 三个维度都 ≥ 4

【Outstanding Pass】
- 三个维度都 ≥ 4，至少两个 5

【No Pass】
- 任一维度 < 2
- 或两个维度 < 3
```

## 常见陷阱（面试官要警惕）

### 1. 只看代码速度，忽视思路

**陷阱**：
```
候选人很快写完，但写的时候一直沉默，
代码也有几个微妙的 bug。
```

**正确做法**：
- 要求候选人解释思路
- 即使速度快，如果 bug 多或无法解释也不应该 pass

### 2. 过度提示导致错误评价

**陷阱**：
```
候选人卡住后，你给了很多提示。
最后他完成了，但其实这些是你的想法。
```

**正确做法**：
- 提示应该是"你考虑过 X 吗？"而不是"应该怎么怎么样"
- 如果需要多轮提示，说明候选人本身能力有限

### 3. 对"没做到"的功能太苛刻

**陷阱**：
```
时间不够，候选人没实现 REDO。
你因此认为他能力不足。
```

**正确做法**：
- 90 分钟本来就紧张
- 评价他做到了什么做得有多好，而不是没做的
- "如果有时间，你会怎么添加 REDO？"来检验思路

### 4. 忽视候选人的学习信号

**陷阱**：
```
候选人一开始有个设计问题，但能很快意识到并改正。
你还是认为他不行。
```

**正确做法**：
- 在 90 分钟内从 bug 中恢复，展示的是成熟度
- 比一开始完美但拒绝反馈要好得多

## 面试结束的 Quick Debrief（30 秒版）

和同事快速同步评价：

```
"他/她的亮点是 ____（Technical/Pragmatism/Learning）。
弱点是 ____。整体上，我觉得他/她是 [Pass/Strong Pass/No Pass]。"
```

**例子**：
```
"他对多机器人碰撞的设计思路清晰，职责划分恰到好处。
但对边界情况的考虑还不够全面。整体 Pass，可能是 mid-level 的选手。"
```

## 最后的话（给面试官）

记住：
- **这不是 coding challenge**，是**协作面试**
- **不是看他/她能多快写代码**，是**看他/她如何思考和沟通**
- **一个能接纳反馈、承认不确定的 mid-level 工程师**往往比一个"我全知道"但难以协作的人更有价值
